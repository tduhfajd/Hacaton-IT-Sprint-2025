# 05 — AI и база знаний

Этот раздел объясняет, как SmartSupport использует GigaChat, Celery workers и статьи базы знаний, чтобы помогать операторам.

## 🧠 Поток AI-анализа

1. **Backend публикует задачу** в RabbitMQ (`tasks.analyze_appeal` или `tasks.generate_response`), когда создаётся обращение или оператор запрашивает новый ответ.
2. **Celery worker** (см. `workers/tasks/`) извлекает задачу и вызывает GigaChat через `gigachat_client.py`.
3. **Результаты записываются** в PostgreSQL:
   - `appeal_analysis` — информация о тональности, категории, приоритете, ключевых словах и кратком резюме.
   - `ai_responses` — готовые черновики ответов для операторов.
4. **Frontend операторов** получает обновления через Socket.io и показывает подсказки вместе с релевантными статьями.

Если требуется вручную перезапустить процесс анализа, повторно вызовите эндпоинты `/api/ai/analyze/:id` или `/api/ai/generate-response/:id`.

## 🗂️ База знаний

- Статьи хранятся в Markdown файлах в каталоге `knowledge-base/` (по категориям: `vodosnabzhenie.md`, `blagoustroistvo.md` и т.д.).
- При деплое статьи загружаются в таблицу `knowledge_base` (или могут читаться напрямую, в зависимости от текущей реализации).
- Добавляя новый материал:
  1. Создайте файл в `knowledge-base/` с понятным именем (латиница, без пробелов).
  2. Используйте заголовок H1, затем секции, факты, инструкции.
  3. Обновите админку (frontend/admin) или выполните импорт в БД, чтобы статья стала доступной для поиска.
- Старайтесь включать теги и категории, чтобы AI легче подбирал релевантные ответы.

## 🔧 Настройка GigaChat

- Шаблон конфигурации: `gigachat-config.example.json`.
- Параметры:
  - `clientId`, `clientSecret`
  - `authKey` (base64 от `clientId:clientSecret`)
  - `scope` (обычно `GIGACHAT_API_PERS`)
- Проверка подключения:

```bash
cd backend
npm run test:gigachat
```

Полная инструкция — в [GIGACHAT_SETUP.md](../GIGACHAT_SETUP.md).

## 📋 Переменные окружения (AI/БЗ)

- `GIGACHAT_CLIENT_ID`, `GIGACHAT_CLIENT_SECRET`, `GIGACHAT_SCOPE`
- `TELEGRAM_BOT_TOKEN` — для канала подачи обращений.
- `RABBITMQ_*`, `REDIS_*`, `POSTGRES_*` — см. `.env.prod`.
- `KNOWLEDGE_BASE_PATH` (если используется) — путь до Markdown статей.

## 🧪 Проверка качества AI

- **Unit-тесты** workers не покрывают GigaChat, поэтому проверка идёт вручную:
  - отправьте тестовое обращение через user UI или Telegram;
  - убедитесь, что анализ появился в операторской панели;
  - проверьте, что черновик ответа и предложенные статьи корректны.
- Список кейсов и результатов смотрите в [TESTING_REPORT.md](../TESTING_REPORT.md).

## 🛠️ Отладка

- Worker не отвечает:
  - `docker logs smart-support-celery-worker --tail 50`
  - Убедитесь, что `celery-beat` жив (управляет периодическими задачами).
- Нет публикации задач:
  - Проверьте подключение backend к RabbitMQ (переменные `RABBITMQ_URL`).
  - Убедитесь, что очередь `smart_support_ai` создана (см. RabbitMQ UI).
- Ответы не используют новые статьи:
  - Перезагрузите админку и убедитесь, что статья попала в таблицу `knowledge_base`.
  - Очистите кэш Redis (если используется кэширование статей).
